Week 2; Information and Social Engineering

The lecture covered areas relating to information security and techniques for social engineering. In this lecture I learned that the fines for a company that breaches the data protection act used to be a maximum if £500,000 but is now 4% of the annual turnover of the company, which means there is a more significant impact for companies such as Apple, as under the new fine, they could lose up to £6 Billion, compared to the measly £500,000 it used to be. This was done so that there was more of a deterrent for larger companies not following robust security guidelines.

I also learned that there are many social engineering techniques, including ‘Tailgating’, which is where a person follows someone closely in order to gain access to a restricted area they shouldn’t.

In the practical session, I learned that each student that attends the University of Portsmouth has their own cloud based virtual machine, hosted by google, which we can use for hosting our website and database projects.

Week 15 - Usability

The usability lecture was quite informative about what not to do in order to try and make something more useable. Largely, this came down to the matter that I should be designing something that is easy and intuitive to use, that I don’t have to provide any sign-posts for users or visitors to follow in order to navigate. In addition to this, the practical was based on looking at how useable other websites are in terms of design, and functionality. While looking at some examples of websites, such as http://www.ericafirment.com/ and http://mariusz.cc/ which are both good looking and intuitive designs when viewed full screen. However, when the window is shrunk or compressed, both sites begin to exhibit a few issues such as overflowing text/images, menu items not fitting on the bars etc. which are all fairly easy fixes (or so I was informed by Jaeck)

After taking some inspiration from these examples, I began to re-style my own website to;

  • Look better
  • Be intuitive and easy to use
  • Have no window scalability issues

To begin with I started modifying the index page to a style that I liked and got everything positioned how I wanted. Once that was done, I began to resize the window and see what was breaking and what I could keep the same. As all of the text sizes were a set size – all the text kept overflowing and breaking / going out of line etc. At first I attempted to adjust the values by making them all %’s – however as I found out, in CSS % means ‘percent of default size’ – so the text would remain a fixed size still no matter what size the window is. After searching for a while I cam across the view width and view height values (vh, vw) which Rich had mentioned before. These values set the text to a certain percentage of the size of the viewport width or height, meaning that this will scale to fit the size of the viewing window -Perfect!

Another issue that I ran into was that the background image was breaking and started to tile in different sized windows, as the aspect ratio was broken. As well as this, on certain sized screens some elements went off of the screen due to the rigid margin properties I had given them. In order to fix this, I had to look into CSS @media queries, which are able to read certain information about the device the page is being viewed on (like screen size and pixel density etc..) By doing this, I was then able to set different styling for certain screen orientations and sizes etc.. thus making the page scale and present itself in a decent manner on all screens. One of the issues I ran into along the way with the CSS queries was that if I had one defined higher up in the style sheet, it was being overwritten by anything I had written bellow for the same element. This can be avoided by putting all media queries at the bottom, however I found that it is possible to use @media queries to decide on loading a specific style sheet, meaning I simply have multiple sheets to do the styling. The issue with this however, is that I would have to recode the same thing multiple times, and it would have an impact on page loading times as there is more data to be downloaded. Therefore, I am sticking with a single, horribly messy, style sheet.
